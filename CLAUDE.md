# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

Agentor orchestrates Claude Code agents that consume work items from a target project's markdown files (backlog, ideas), develop each item in its own git worktree, and ping the user for review+approval before committing. It is configured **per target project** — agentor itself is the orchestrator, the work happens in some other repo pointed at by `[project].root` in the config.

## Architecture

Pipeline: **watched markdown files → extracted Items → SQLite queue → daemon dispatches to agent pool → per-item worktree → plan phase → awaiting_plan_review → execute phase → awaiting_review → user approve/reject → commit (and optionally merge) → cleanup**.

Key modules:

- `agentor/models.py` — `Item` (immutable, sha1 of `source_file+title+body`) and `ItemStatus` lifecycle: `queued → working → awaiting_plan_review → working → awaiting_review → merged | rejected | errored | conflicted | cancelled | deferred`. The legacy `backlog` enum value was removed; `Store._migrate` heals any stale `'backlog'` strings on open.
- `agentor/config.py` — loads `agentor.toml`. Project root resolves relative to the config file's directory unless absolute. Knobs worth knowing: `agent.pool_size` (caps concurrent `working` items; default 0 — operator bumps via `+` in the dashboard), `agent.runner` (`stub` | `claude` | `codex`), `agent.single_phase` (skip the plan phase), `sources.watch` (glob list), `parsing.mode` (`checkbox` | `heading` | `frontmatter`), `git.merge_mode` (`merge` | `rebase`).
- `agentor/extract.py` — parses markdown into Items. Modes:
  - **checkbox**: each `- [ ]` is an item; continuation lines form the body; `- [x]` skipped.
  - **heading**: each `#`..`######` is an item; body runs until the next heading of same-or-higher level.
  - **frontmatter**: one item per file; title/state/tags taken from YAML frontmatter.
  - Inline `@key:value` tags are stripped from title/body into `item.tags`. Title tags win on conflict.
- `agentor/store.py` — SQLite state store. Atomic `claim_next_queued`, `pool_has_slot`, history via the `transitions` table, failure rows via `failures`. All transitions go through `Store.transition(to, **fields)`.
- `agentor/watcher.py` — `scan_once` diffs the markdown-extracted items against the DB and inserts new ones at QUEUED.
- `agentor/daemon.py` — main loop. Polls the watcher, dispatches queued items into a thread pool capped by `pool_size`, surfaces infra failures as sticky alerts on the dashboard until the user presses `u`.
- `agentor/runner.py` — `Runner` base class + `StubRunner`, `ClaudeRunner`, `CodexRunner`. Two-phase flow: `plan` (read-only; stops at AWAITING_PLAN_REVIEW) → human approves via `approve_plan` → `execute` resumes the same session (`--resume <session_id>` for claude, `thread_id` for codex) and commits. `single_phase=true` skips plan and goes straight to execute.
- `agentor/committer.py` — handles AWAITING_REVIEW → MERGED. `approve_and_commit` commits any uncommitted work on the feature branch, then integrates into `git.base_branch` via an ephemeral detached worktree (`merge` → `--no-ff`, `rebase` → `rebase <base>`-then-CAS-fast-forward). On conflict the item transitions to CONFLICTED with the summary in `last_error`, then is unconditionally chained through `resubmit_conflicted(force_execute=True)` so the agent is re-queued to resolve the merge itself without operator intervention (force_execute flips `result_json["phase"]` so the runner skips the plan phase — conflict resolution is pure execute work). Covers every failure mode `merge_feature_into_base` funnels into CONFLICTED — true merge/rebase conflicts, rebase aborts, and CAS races. `retry_merge` is still available from the dashboard `[m]` key for operators who prefer to resolve by hand.
- `agentor/recovery.py` — runs at daemon startup. WORKING items with a live `session_id` + worktree go back into resumable; everything else reverts to its previous settled status. Also clears benign stale `last_error` markers.
- `agentor/dashboard/` — curses UI package. `render.py` owns screens/prompts, `modes.py` owns actions (review/deferred/inspect), `formatters.py` formats table cells, `transcript.py` parses the claude stream-json session feed. Main table auto-refreshes at `REFRESH_MS=500`; modes: review `r`, deferred `d`, inspect `i`. Inspect auto-refreshes every 1s; `[m]` retries merge for CONFLICTED items.

Design invariants to preserve:

- **Item IDs must be stable across runs** — the daemon diffs parsed items against SQLite to detect new/removed work. Changing the hash input breaks deduplication.
- **Path resolution**: `extract_items` calls `.resolve()` on both the source file and project root before `relative_to` to handle macOS `/tmp` → `/private/tmp` symlinks. Don't drop this.
- **Agent pool is enforced at the scheduler, not in config**: pool_size just caps `COUNT(status='working')`. Bumping the number requires no code changes.
- **Per-project scope**: agentor runs against one project dir; it never spans multiple repos in a single run.
- **Worktrees start from the current tip of `git.base_branch`** — `worktree_add` passes the branch name, so git resolves the sha at dispatch time. Resumed worktrees (plan → review → execute) run `fast_forward_to_base` before `do_work` to pull in any base-branch commits that landed during the review gap; if the feature has diverged (agent committed during plan), ff refuses and we fall through silently so the final integration step handles the divergence.
- **Auto-merge integrates via a detached temp worktree** — `merge_feature_into_base` always works in a `--detach`ed temp worktree and CAS-advances `refs/heads/<base>` via `update-ref OLD NEW`. The user's checkout at `project.root` is never used as a workspace for the merge itself. After a clean success the committer optionally fast-forwards the user's checkout to the new tip via `advance_user_checkout`, but only when (a) the checkout is on `base_branch`, (b) the working tree is clean, and (c) HEAD equals the pre-merge base sha — otherwise it leaves the checkout alone. The outcome is recorded on the MERGED transition note as either `, checkout advanced` or `, checkout skipped: <reason>` (reason one of `detached HEAD`, `checkout on <branch>`, `dirty worktree`, `HEAD diverged from pre-merge base`) so operators can grep history for the skip cause; gate-off stays silent (no suffix). Operators who deliberately park another branch at `project.root` opt out via `git.advance_user_checkout = false`.
- **Feedback is consumed once**: the runner's `_prepend_feedback` reads `item.feedback`, injects it into the next prompt, and clears the column so a future run starts clean.
- **Plan can nominate the execute-phase model tier** — the plan's `## Execute tier` trailer (`suggested_model: haiku|sonnet|opus`) is parsed by `runner._parse_execute_tier` and resolved on execute dispatch via `_resolve_execute_tier`. Precedence (first match wins): item `@model:<alias>` tag > plan suggestion (only when `agent.auto_execute_model=true`) > `agent.model`. Whitelist gate (`agent.execute_model_whitelist`) rejects anything outside the configured aliases — invalid/missing is a *soft fallback with a warning*, never a hard error (legacy plans in `transitions` history carry no trailer, so a raise would make them un-resumable). Resolved `(alias, source)` lands on `result_json["execute_model"]` / `result_json["execute_model_source"]` for post-hoc cost attribution. `single_phase=true` skips the plan → no suggestion possible → always falls through to tag-or-default. `_ALIAS_TO_MODEL` in `agentor/config.py` is the single place alias → current best model id lives; rotate it in lockstep with Anthropic releases. Custom `agent.command` overrides that drop the `{model}` placeholder silently skip the per-invocation override (same opt-out pattern as `{settings_path}`). The dashboard `[M]` picker DOES NOT touch model tier — it flips provider only (see next bullet); model tier is driven exclusively by `@model` tag / plan nomination / `agent.model`.
- **Dashboard `[M]` picker swaps provider, not model** — the shift-M overlay writes a runner kind (`claude` | `codex`) into `daemon.provider_override`, which `daemon._make_runner` reads via `dataclasses.replace` to shadow `config.agent.runner` for the NEXT fresh dispatch. Already-handed-off workers are unaffected (the runner instance's type is frozen at construction). AWAITING_PLAN_REVIEW items re-enter via QUEUED on `approve_plan`, so a mid-session flip DOES re-target their execute phase. Override is in-memory only, wiped on daemon restart, never written to `agentor.toml`. `stub` is deliberately excluded from `PROVIDERS` — it's a test fixture, not a user-toggled provider.

## Agent commit rules

Hard rules every agent run must satisfy before its work is approvable. These are enforced (where possible) at the merge gate, but stating them up front saves a CONFLICTED roundtrip.

- **Delete the source backlog markdown in the SAME commit as the implementation.** Every queued item that came from `docs/backlog/<slug>.md` MUST `git rm` (or equivalent delete + `git add`) that file as part of the final implementation commit — not a trailing cleanup commit, not a follow-up PR. If the file is genuinely absent at dispatch (already deleted by a prior scan, or was an untracked scratch file), note the deviation in the commit body — don't fabricate a phantom delete. Stale backlog files accumulate fast and pollute `scan_once`'s diff against the DB; periodic operator-driven sweeps to prune them is a smell, not a workflow.
- **Write a per-run findings file under `docs/agent-logs/<YYYY-MM-DD>-<slug>.md`.** Captures Surprises / Gotchas / What worked. Enforced softly by `committer.approve_and_commit` (appends `, no agent-log written` to the MERGED note when missing); upgrade to a hard CONFLICTED block via `agent.require_agent_log = true`. The fold pipeline (`agent.fold_threshold`) folds these into CLAUDE.md gotchas in batches.

## Gotchas from prior runs

Hard-won traps future agents keep rediscovering. Durable lessons get folded here from `docs/agent-logs/` via an auto-queued backlog item: the daemon notices when `docs/agent-logs/` crosses `agent.fold_threshold` files (default 10) and drops `docs/backlog/fold-agent-lessons-YYYY-MM-DD.md` so a future agent clusters recurring Surprises / Gotchas into this file and deletes the consumed logs in the same commit. The CLAUDE.md diff still goes through the normal review gate — no auto-merge.

- **Stale `file:line` refs in backlog items** — the codebase moves; `agentor/dashboard.py` is now the `agentor/dashboard/` package. Grep stable symbols, not filenames, before trusting any path cited in a ticket.
- **`last_error` is capped at 4000 chars** — `Store.transition` stores whatever you hand it, but the inspect view and downstream consumers truncate at 4000. When composing structured summaries (see `committer._build_conflict_summary`), pre-cap each section independently (`_BODY_CAP=2000`, `_RAW_CAP=1500`) so the trailing mechanics block survives; a single outer `[:4000]` silently amputates whatever came last.
- **Status SQLite boundary is centralized** — `_encode_status` / `_decode_status` in `store.py` are the only place `ItemStatus` ↔ string crosses the DB. Don't sprinkle `.value` at new callsites; route through the helpers. Also: `Store.transitions_for` returns `Transition` dataclasses with `from_status`, `to_status`, `note`, `at` — not dicts, and the status fields are already decoded enums.
- **`tools/` scripts run as files, not modules** — importing from `agentor` requires `sys.path.insert(0, str(Path(__file__).resolve().parent.parent))` at the top (see `tools/analyze_transcripts.py`). Don't add a `__init__.py` to `tools/` — keep scripts runnable as `python3 tools/foo.py` from the repo root.
- **Lazy `..committer` imports in `dashboard/modes.py`** — action functions do `from ..committer import …` inside the function body, not at module top, to sidestep a circular import via `store`. Preserve the pattern when wiring new actions; a top-level import will break dashboard startup.
- **Python version pinning is load-bearing** — `requires-python = ">=3.11"`. Nested f-strings reusing the same quote char are PEP 701 (3.12+) and will `SyntaxError` on 3.11 CI. Keep ruff `target-version = "py311"` and mypy `python_version = "3.11"` in `pyproject.toml` so devs on 3.13 get the error locally instead of in CI.
- **Merge shortcut for cosmetic-vs-semantic conflicts** — when `main` lands a cosmetic refactor (reorders, banners, rename sweeps) while the branch edits the same functions, prefer `git checkout --theirs -- <file>` then manually re-apply the semantic deltas. Git's recursive merge has silently corrupted `store.py` past the conflict markers in the past; trust the three-way resolution less than your own re-application of a small diff.
- **Dashboard hot paths must not do O(file) or O(history) work per tick** — inspect view refreshes once a second, the main table every 500ms. Transcript reads use `iter_events(..., tail_bytes=_TAIL_BYTES)` (see `dashboard/transcript.py`) — never `path.read_text()`. Elapsed-since-working uses `Store.latest_transition_at` — never `transitions_for` + scan. Past hang reports traced to whole-file transcript reads starving `getch` on multi-MB live logs.
- **Headless curses tests must patch color/napms at the resolved module** — `_render` / `_render_table` call `curses.color_pair(...)` and `_flash` calls `curses.napms(...)`, both of which error outside an `initscr()` context. Patch `agentor.dashboard.render.curses.color_pair` (return `0`) and `agentor.dashboard.render.curses.napms` — `render` is where the symbols resolve, not `modes`, even when the caller lives in `modes.py`. `tests/test_dashboard_render.py::TestPriorityGlyph` is the working template.
- **Backlog source markdown may be absent at dispatch** — the mandatory `git rm docs/backlog/<slug>.md` step is best-effort. A prior scan may have extracted+promoted the item (deleting the file earlier), or the source may have been an untracked scratch file in the parent repo that git never tracked. If the file isn't in the worktree, skip the `git rm` and note the deviation in the commit body — don't fabricate a phantom delete.
- **Main races ahead during plan/review gaps** — hot dashboard files (`__init__.py`, `render.py`, `modes.py`, `committer.py`, `store.py`) attract keybinding/layout edits on every round-trip, so expect merge conflicts on each resumed execute. Pre-execute: `git merge main` and re-run tests before re-planning so you inherit any newly-merged plumbing (example: `write_claude_settings` + `--settings` wiring landed mid-plan and made the branch's parallel settings path obsolete). When `main` lands your feature concurrently, accept `--theirs` in full and delete your branch's now-obsolete scaffolding rather than preserving both designs.
- **Removing an `ItemStatus` member must heal all three status columns in `_migrate`** — `items.status` AND `transitions.from_status` AND `transitions.to_status`. Migrations that touch only `items` leave `transitions_for` ready to `ValueError` on the next read of any legacy-history row. Also sweep `agentor/fold.py:_NON_TERMINAL_STATUSES` and any enum-keyed dict with hardcoded members (`_STATE_GLYPHS` in `dashboard/render.py` has silently held stale `ItemStatus.BACKLOG` keys past enum removal — mypy won't flag it because the dict type is `dict[ItemStatus, str]`). Prefer loops over `ItemStatus` members so the next rename surfaces at type-check time.
- **Curses `KEY_RESIZE` must be handled in every non-`_render` getch loop** — inspect, help, diff scroller, progress overlay, and `_prompt_multiline`'s `Textbox.edit` validator each need their own branch; `_handle_resize(stdscr, ch)` centralises it. After a resize, use `stdscr.clear()` (not `erase()`) to force a full repaint — `erase()` blanks the internal buffer but the diff-against-terminal-cache can come up empty and leave the previous wide-tier frame on screen, pushing row 0 off. Guard `curses.update_lines_cols()` with `hasattr` for Windows-curses compatibility.
- **`curses.textpad.Textbox` defaults bite multi-line editors** — `stripspaces=True` (default) silently eats trailing whitespace on every line, including blank paragraph separators; set `stripspaces = False` after construction. Remap Ctrl-C (keycode 3) inside the validator so `KeyboardInterrupt` can't leak out of the dashboard loop. `_prompt_multiline` in `dashboard/render.py` is the reference implementation.
- **Action-bar strings and key-maps are test contracts** — `_ACTION_KEYS_BY_STATUS` in `dashboard/modes.py` is pinned by `tests/test_dashboard_enter.py` as exact sets per status; `ACTIONS` / `ACTIONS_WIDE` / `ACTIONS_MID` tokens are pinned by `tests/test_dashboard_render.py::TestActionsHint.test_core_actions_present`. Adding or removing a key means updating both the table/string AND the matching regression test — ACTIONS drift is the canonical merge-conflict signal when two branches both edit keybindings.
- **Stream-json transcript & subprocess quirks** — transcript filename derives from `phase_tag = "execute" if had_session else "plan"` in `_invoke_claude`, so `single_phase=True`'s first run still writes `<id>.plan.log` (tests asserting against transcript paths must derive the same way, not from the semantic plan/execute boundary). Fake-CLI shell scripts that `read foo` from runner stdin must use `read foo || true` so the script exits 0 once the runner closes stdin — `/bin/sh read` returns non-zero on EOF and flakes assertions on `exit: 0`. Retry wrappers appending RETRY markers must clear the transcript **once** in the outer dispatch; inner writers append only — `_run_stream_json_subprocess` and `_invoke_claude_blocking` both previously `write_text`'d per invocation.
- **`advance_user_checkout_allowed` guards evaluate BEFORE CAS, never after** — once `merge_feature_into_base` advances `refs/heads/<base>` via `update-ref`, `HEAD` at `project.root` symbolically follows the new ref, so post-CAS `rev-parse HEAD` returns the new sha and `status --porcelain` shows every file in the base diff as a spurious staged change. Capture `base_sha_before` and the cleanliness verdict pre-CAS. The advance itself uses `git reset --hard <sha>`, not `git merge --ff-only`, because `--ff-only` fires the `post-merge` hook while `reset` is hook-free. The dirty check passes `--untracked-files=no` to `git status --porcelain` because `git reset --hard` preserves untracked files — operators routinely have untracked `docs/backlog/*.md` drops sitting in the base checkout while the daemon runs, and treating those as dirty would lock everyone out of every merge. Valid reason suffixes / block markers: `detached HEAD`, `checkout on <branch>`, `HEAD diverged from pre-merge base` (all soft skips, MERGED with `, checkout skipped: <reason>`), and `dirty worktree` (HARD BLOCK — see next bullet).
- **Dirty base-branch checkout is a HARD merge block, not a skip** — `committer._decide_checkout_advance` returns `("block", …)` when `advance_user_checkout_allowed` reports `dirty worktree`. `approve_and_commit` and `retry_merge` both transition the item to CONFLICTED with `last_error` starting with `Refusing to merge` and the `git status --porcelain --untracked-files=no` output appended, then return early WITHOUT entering the auto-resolve chain (the agent can't fix the operator's dirty checkout). Operator must clean main checkout (commit / stash / discard) and re-trigger via `[m]` retry-merge. Other guard reasons (detached HEAD, parked on another branch) remain soft skips since the operator deliberately opted out of advance for those cases. Gate is gated by `git.advance_user_checkout` — opting out also disables the block.
- **Hard-delete + tombstone coupling** — `Store.delete_item` (driven by the `[x]delete` action) physically removes rows and writes a `deletions` tombstone. `Store.upsert_discovered` must consult the tombstone or `scan_once` immediately re-enqueues just-deleted ids whose source markdown still carries the entry. `PRAGMA foreign_keys = ON` is set on every connection, so DELETE order (dependents first: failures → transitions → items) is load-bearing. Hard-delete of a WORKING row may race the runner's post-kill `transition(ERRORED, …)` or `note_infra_failure`, raising `KeyError`; `Daemon._run_worker`'s outer `except Exception` swallows it as a `worker crashed: <id>: <err>` log — noisy but non-fatal. Tests exercising the WORKING-teardown delete path must patch `agentor.committer._DELETE_WAIT_SECONDS` or each subTest stalls ~5s polling a stub that never exits WORKING.
- **Token-windows aggregator is a TTL cache** — `_token_windows` in `dashboard/formatters.py` caches the per-window `aggregate_token_usage` result for 2s keyed on `(id(store), daemon_started_at)` because the uncached path does a full `items` scan + `json.loads` on every `result_json` blob per 500ms tick. Tests that seed `result_json` and then call `_render` must call `_token_windows_invalidate()` in `setUp`, or a prior test's cached aggregate masks the fresh one. `_render_token_panel` takes `tier` positionally and `windows` as a kwarg — keep the shared pre-computed windows threaded through to avoid re-hitting the cache twice per tick.
- **Lazy `..committer` imports extend to module-level constants, not just action functions** — a top-level `from ..committer import X` in `dashboard/modes.py` is a landmine even when `X` is a plain string constant, because `committer` imports `store` which can re-enter dashboard-adjacent code. Lazy-import inside the consumer (the existing rule for action functions applies to constants too).
- **Serialized integration via `_integration_lock(repo)`** — wraps the module-level `_INTEGRATION_LOCK` (intra-process) AND an `fcntl.flock` on `<repo>/.agentor/merge.lock` (cross-process) so neither two threads in one daemon nor two daemon processes pointed at the same repo can race on `update-ref`. Any new caller of `git_ops.merge_feature_into_base` must either route through `committer.approve_and_commit` / `retry_merge` or wrap its own call in `_integration_lock(repo)` — bare `_INTEGRATION_LOCK.acquire()` is the cross-process race come back. `fcntl` makes this Unix-only; matches the curses dashboard's existing platform constraint.
- **Per-run findings log is checked at the merge gate** — `committer.approve_and_commit` diffs the feature branch against `git.base_branch` and looks for at least one added file under `docs/agent-logs/*.md` (see `git_ops.added_agent_logs`). Default knob (`agent.require_agent_log = false`): a miss only appends `, no agent-log written` to the MERGED transition note, so operators can grep history for the skip rate (`git log`/`store.transitions_for`). Opt-in `agent.require_agent_log = true` upgrades the miss to a CONFLICTED block with `last_error = "agent-log missing"` — the item is re-queued via `resubmit_conflicted` (agent generates the log) or resolved via the normal retry path once the operator adds one. The gate only runs in `approve_and_commit`; `retry_merge` does not re-check, so a manual fix followed by `[m]` produces a MERGED note without the skip marker.

## No-deps policy

Stdlib only (`tomllib`, `sqlite3`, `pathlib`, `hashlib`, `re`, `curses`, `threading`, `json`). Adding a third-party dep should be a considered decision — the current shape works without any.
